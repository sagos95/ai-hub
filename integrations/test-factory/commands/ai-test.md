# AI Test — автоматическое создание тестов с валидацией качества

Создаёт тесты для описанного функционала по Ralph-паттерну: кодо-агент пишет тест → ревьювер-агент проверяет → цикл до 5 итераций.

## Константы

```
MAX_ITERATIONS = 5
REPOS_DIR_FILE = .repos-dir
```

## Входные данные

Аргумент может быть:
- **Путь к локальному репозиторию** (абсолютный путь) — опционально с описанием через ` — `
- **Kaiten card_id** (числовой) — описание и affected services берутся из карточки
- **Текстовое описание** функционала — спросит путь к коду интерактивно

## Workflow

### Фаза 1: Сбор контекста

#### 1a: Если передан путь к репозиторию

Аргумент содержит абсолютный путь (начинается с `/` или `~/`). Опциональное описание после ` — `:
```
/ai-hub:ai-test "/path/to/your-service — retry logic in HttpClient on timeout"
/ai-hub:ai-test "/path/to/your-service"
```

1. Определи git-состояние репозитория:
   ```bash
   git -C <repo_path> branch --show-current
   git -C <repo_path> log main..HEAD --oneline
   git -C <repo_path> diff main...HEAD --name-only
   ```

2. Из этих данных извлеки:
   - **Текущая ветка** — на ней и будет работать test-coder
   - **Изменённые файлы** — что именно было затронуто
   - **Коммит-сообщения** — дополнительный контекст о сути изменений

3. `FUNCTIONALITY_DESCRIPTION`:
   - Если описание передано после ` — ` → используй его
   - Если нет → сформулируй из коммит-сообщений и diff

Интерактивные вопросы **не задавай**, если путь передан. Переходи к Фазе 1d.

#### 1b: Если передан Kaiten card_id

1. Получи карточку:
   ```bash
   ./integrations/kaiten/scripts/kaiten-cards.sh get <card_id>
   ```
2. Извлеки описание, чек-листы, affected services
3. Получи affected services:
   ```bash
   # Extract affected services from card custom fields if available
   ```
4. Определи путь к репозиториям (прочитай `.repos-dir` или спроси пользователя)
5. Проверь, что репозитории доступны локально
6. Если возможно, определи git-состояние (ветку, изменённые файлы) как в шаге 1a

#### 1c: Если передано текстовое описание

1. Спроси у пользователя:
   - **Путь к исходному коду** — где находится репозиторий?
   - **Какой функционал тестируем?** — уточни, если описание неоднозначно
2. Если возможно, определи git-состояние как в шаге 1a

#### 1d: Составь спецификацию

На основе собранной информации сформируй чёткую спецификацию:

```
FUNCTIONALITY_DESCRIPTION = <что именно нужно протестировать>
REPO_PATH = <абсолютный путь к репозиторию>
CHANGED_FILES = <если есть, файлы из git diff main...HEAD>
LANGUAGE = <определи по расширениям изменённых файлов и структуре проекта>
TEST_FRAMEWORK = <определи по существующим тестам в проекте>
```

### Фаза 2: Оценка тестируемости

Прежде чем запускать цикл, оцени — **можно ли написать достоверный автоматический тест** для данного функционала.

**НЕ тестируемые сценарии** (сообщи пользователю и ОСТАНОВИСЬ):
- Визуальные эффекты: анимации, CSS-переходы, pixel-perfect вёрстка
- Субъективные оценки: "выглядит красиво", "удобный UX", "текст понятен"
- Внешние зависимости без эмуляции: живые сторонние API без sandbox/mock
- Hardware-зависимое поведение: NFC, камера, Bluetooth, реальные принтеры
- Недетерминированные процессы: результат зависит от timing, race conditions (если нет способа сделать детерминированным)

Если функционал НЕ тестируем, выведи:

```
⛔ Тест не может быть написан

Функционал: <описание>
Причина: <почему невозможен достоверный автоматический тест>
Рекомендация: <как можно проверить вручную или альтернативный подход>
```

**Тестируемые сценарии** (продолжай к Фазе 3):
- Бизнес-логика: расчёты, валидации, трансформации данных
- API-контракты: запросы/ответы, статус-коды, ошибки
- Интеграции: взаимодействие компонентов (с моками внешних систем)
- State management: переходы состояний, обработка событий
- Data processing: парсинг, сериализация, маппинг
- Конфигурации: правильная загрузка, fallback-значения
- Сервисная логика: обработка ошибок, retry, circuit breaker

### Фаза 3: Ralph-цикл (Coder → Reviewer)

```
iteration = 0
reviewer_feedback = ""
status = "IN_PROGRESS"
```

**Запусти цикл `while iteration < MAX_ITERATIONS && status == "IN_PROGRESS"`:**

#### Шаг 3.1: Запуск test-coder агента

Запусти агента `test-coder` через **Task tool** с чистым контекстом:

```
Ты — test-coder агент. Итерация: <iteration + 1> из <MAX_ITERATIONS>.

## Задача
Создай автоматический тест, который достоверно проверяет следующий функционал:

<FUNCTIONALITY_DESCRIPTION>

## Репозиторий
Путь: <REPO_PATH>

## Изменённые файлы (что тестировать)
<CHANGED_FILES>

## Язык и фреймворк
- Язык: <LANGUAGE>
- Тест-фреймворк: <TEST_FRAMEWORK, если известен; иначе "определи по проекту">

<ЕСЛИ iteration > 0:>
## Фидбек ревьювера (ОБЯЗАТЕЛЬНО учти!)
Предыдущая версия теста была отклонена ревьювером. Исправь ВСЕ указанные проблемы:

<reviewer_feedback>

## Предыдущий тест (для reference)
Файл: <test_file_path>
<содержимое предыдущего теста>
</ЕСЛИ>

## Инструкции
1. Исследуй тестовую инфраструктуру проекта
2. Изучи исходный код тестируемого функционала, изменённые файлы и пойми что тестировать
3. Напиши тест, следуя конвенциям проекта
4. Запусти тест и убедись, что он проходит
5. Верни структурированный отчёт
```

Сохрани результат: `coder_report`, `test_file_path`, `test_status`.

Если coder вернул статус ❌ (не смог создать тест):
- Если причина — отсутствие тестовой инфраструктуры или технический блокер — **остановись** и сообщи пользователю
- Если причина — сложность задачи — позволь ещё одну итерацию

#### Шаг 3.2: Запуск test-reviewer агента

Запусти агента `test-reviewer` через **Task tool** с чистым контекстом:

```
Ты — test-reviewer агент. Твоя задача — независимо оценить качество теста.

## Описание тестируемого функционала
<FUNCTIONALITY_DESCRIPTION>

## Репозиторий
Путь: <REPO_PATH>

## Изменённые файлы (что тестируется)
<CHANGED_FILES>

## Тест для ревью
Файл: <test_file_path>

## Результат прогона теста
<test run output из отчёта coder-а>

## Инструкции
1. Прочитай описание функционала
2. Прочитай исходный код (изменённые файлы в репозитории)
3. Прочитай тест
4. Оцени: тест РЕАЛЬНО проверяет описанный функционал или является фикцией?
5. Верни вердикт: APPROVED или NEEDS_REVISION с конкретным фидбеком
```

Сохрани результат: `reviewer_verdict`, `reviewer_feedback`.

#### Шаг 3.3: Решение оркестратора

- Если `reviewer_verdict == "APPROVED"`:
  ```
  status = "APPROVED"
  → выход из цикла
  ```

- Если `reviewer_verdict == "NEEDS_REVISION"`:
  ```
  iteration += 1
  → продолжить цикл с feedback
  ```

**После каждой итерации выведи промежуточный статус:**

```
🔄 Итерация <N>/<MAX_ITERATIONS>
📝 Coder: <краткий статус — создал/обновил тест>
🔍 Reviewer: <APPROVED/NEEDS_REVISION — ключевое замечание>
```

### Фаза 4: Результат

#### Если APPROVED:

```
✅ Тест создан и одобрен

📄 Файл: <test_file_path>
🧪 Фреймворк: <framework>
▶️ Запуск: <run command>
🔄 Итераций: <iteration + 1>/<MAX_ITERATIONS>

📋 Тесты:
- `TestMethod1` — что проверяет
- `TestMethod2` — что проверяет

✅ Ревьювер подтвердил: тест достоверно проверяет описанный функционал.
```

#### Если MAX_ITERATIONS исчерпан:

```
⚠️ Максимальное количество попыток (<MAX_ITERATIONS>) исчерпано

📄 Последний тест: <test_file_path>
🔄 Итераций: <MAX_ITERATIONS>/<MAX_ITERATIONS>
❌ Последний фидбек ревьювера:
<reviewer_feedback>

Рекомендация: проверьте тест вручную и решите, приемлем ли он в текущем виде.
```

## Важно

- Каждый агент (coder и reviewer) запускается в **новой сессии** через Task tool — чистый контекст, никакой предвзятости
- Reviewer **никогда не видит** процесс создания теста — только результат
- Coder **не знает** кто будет ревьюить — пишет для абстрактного ревьювера
- Оркестратор **не модифицирует код** — только координирует агентов и принимает решения
- При передаче фидбека coder-у **не вырезай критику** — передавай feedback полностью
- Тест должен проверять **существующий код** — если функционал ещё не реализован, это не задача для ai-test
