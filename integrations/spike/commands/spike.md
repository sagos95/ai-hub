# Spike Research Workflow

Выполни spike-исследование для карточки Kaiten.

## Входные данные

- `card_id` — ID карточки в Kaiten (числовой)

## Workflow

### Контекст организации

Для понимания ландшафта сервисов, зависимостей и стека — загрузи `ORG-SERVICES-INDEX.md` из корня проекта (если есть).

### Фаза 1: Discovery

1. Получи карточку из Kaiten:
   ```bash
   ./integrations/kaiten/scripts/kaiten-cards.sh get <card_id>
   ```

2. Извлеки из JSON карточки:
   - Название задачи
   - Описание проблемы
   - Affected services (из custom_fields или тегов, если есть)
   - **Связанные карточки** (parents, children, relations) — запомни, есть ли они. Если карточка спайка не связана ни с одной другой — это будет учтено в рекомендуемых действиях (секция 9).

3. Сформулируй research question в формате "How might we...?"

4. Создай spike-файл `Spikes/YYYY-MM-DD_<name-kebab>_<card_id>.md` с шаблоном:

```markdown
# Spike: <название>

> 🎫 **Kaiten:** [#<card_id>](https://YOUR_DOMAIN.kaiten.ru/space/YOUR_SPACE_ID/card/<card_id>)
> 📅 **Дата:** YYYY-MM-DD
> 🔄 **Статус:** 🔄 In Progress

## Executive Summary

_Заполняется после завершения исследования._

> **Контекст:** <кто пожаловался / попросил, на что, почему — в 1-2 предложениях>
>
> **Класс сложности:** <банальная настройка | небольшая доработка | крупная доработка, несколько вариантов | требует дополнительного исследования>
>
> **Результат:** <структурно: что и где нужно сделать, например "admin-panel → меню X, опцию Y включить" или "доработка в service-A + service-B">
>
> **Вердикт:** <краткий итог в 1-2 предложениях — что делать дальше>
```

### Фаза 2: Setup

1. Используй affected services, извлечённые из карточки в Фазе 1.
   Если affected services не указаны — спроси у пользователя, какие сервисы затронуты.

2. **Определи путь к репозиториям:**

   a. Проверь локальный конфиг `.repos-dir` (в корне проекта). Он содержит путь к папке с уже выкачанными репозиториями (например, `/Users/vasya/my-repos`).
   ```bash
   REPOS_DIR=$(cat .repos-dir 2>/dev/null)
   ```

   b. Если конфиг существует — проверь, есть ли нужный репозиторий локально:
   ```bash
   ls "$REPOS_DIR/<repo-name>" 2>/dev/null
   ```

   c. Если репозиторий найден локально — используй его напрямую (путь `$REPOS_DIR/<repo-name>`). **Не клонируй повторно.**

   d. Если конфиг отсутствует или репозитория нет локально — найди его в GitHub и клонируй в `Temp/`:
   ```bash
   gh repo list your-org --json name --limit 100 | jq -r '.[].name'
   ```

   **ОБЯЗАТЕЛЬНО спроси подтверждение у пользователя** перед клонированием:
   ```
   Репозиторий `your-org/<repo-name>` не найден локально.
   Клонировать в `Temp/`? Продолжаем?
   ```

   После подтверждения:
   ```bash
   cd Temp && git clone --depth 1 https://github.com/your-org/<repo-name>.git
   ```

3. Запомни итоговый путь к каждому репозиторию (локальный или `Temp/`) — он будет использоваться в Фазе 3.

### Фаза 3: Research

Запусти агента `spike-explorer` для исследования кода:

```
Исследуй кодовую базу по теме: <research question>
Репозиторий: <путь к репозиторию из Фазы 2>
Spike-файл: <полный путь к spike-файлу>
Секция: "### 4.1 Кодовая база"

Замени placeholder "_Ожидает исследования..._" на результаты.
Верни ТОЛЬКО краткий статус (5-7 строк).
```

Затем запусти агента `spike-researcher` для поиска best practices:

```
Найди best practices по теме: <research question>
Контекст: C#, .NET, микросервисы (или другой стек)
Spike-файл: <полный путь к spike-файлу>
Секция: "### 4.2 Best Practices"

Замени placeholder "_Ожидает исследования..._" на результаты.
Верни ТОЛЬКО краткий статус (5-7 строк).
```

### Фаза 4: Analysis

Запусти агента `spike-architect` для формирования решения:

```
Проанализируй findings и сформируй решение.
Spike-файл: <полный путь к spike-файлу>
Секции 4.1 и 4.2 содержат исследование.
Секции для записи: "## 5. Решение", "## 6. Блокеры", "## 9. Рекомендуемые действия"

Замени placeholders на решение, блокеры и рекомендуемые действия.
Выяви все блокеры (технические и бизнесовые), которые помешают разработчику выполнить задачу за один спринт.
Сформируй организационные рекомендации (какие задачи создать/обновить/закрыть в Kaiten).
Верни ТОЛЬКО краткий статус (7-10 строк).
```

### Фаза 5: Summary

1. Обнови статус в spike-файле на `✅ Completed`

2. **Заполни Executive Summary** в начале файла (замени placeholder). Это самая важная секция — она должна экономить время читателя:
   - **Контекст:** кто пожаловался/попросил, на что, почему, зачем — из описания карточки Kaiten
   - **Класс сложности:** оцени по результатам исследования (банальная настройка / небольшая доработка / крупная доработка / требует дописследования)
   - **Результат:** структурно опиши, что и где надо делать. Конкретные сервисы, настройки, действия. Например: `admin-panel → меню X, включить опцию Y` или `доработка в service-A + service-B: добавить endpoint Z`
   - **Вердикт:** если решение банальное — так и напиши в двух словах. Если сложное и неочевидное — укажи "крупная доработка, есть N вариантов, детали в секции 5"

3. Заполни секцию "7. Next Steps" конкретными действиями

4. Заполни секцию "8. Open Questions" оставшимися вопросами

5. **Проверь секцию "9. Рекомендуемые действия"** — убедись, что architect заполнил её. Дополни при необходимости:
   - Всегда должен быть пункт `[human-review]` — "Артефакт спайка проверен человеком"
   - **Связанность карточки:** если в Фазе 1 обнаружено, что у карточки спайка нет связанных карточек (ни родительских, ни дочерних) — добавь рекомендацию `[link-task]` с конкретным указанием, например: "Прилинковать карточку спайка к родительской задаче, так как карточка не связана ни с одной другой". Если по результатам спайка рекомендуется создать задачу на разработку — добавь `[link-task]` для линковки спайка с будущей задачей на разработку.

6. **Запуши артефакт спайка в GitHub** — это единственный source of truth:
   ```bash
   git add Spikes/<spike-file>.md
   git commit -m "Add spike: <название>"
   git push
   ```

7. **Добавь ссылку на артефакт в Kaiten-карточку** (комментарием или в описание):
   ```
   Результат спайка: https://github.com/sagos95/ai-hub/blob/main/Spikes/<spike-file>.md
   Краткий итог: <1-2 предложения из Executive Summary>
   ```
   **НЕ загружай .md файл в карточку** — source of truth только GitHub.

8. **Добавь чек-пункт в Kaiten-карточку спайка** — "Артефакт спайка проверен человеком":
   ```bash
   # Получи существующие чек-листы
   ./integrations/kaiten/scripts/kaiten-cards.sh checklists <card_id>
   # Если чек-листа нет — создай
   ./integrations/kaiten/scripts/kaiten-cards.sh checklist <card_id> "Spike checklist"
   # Добавь пункт (используй checklist_id из предыдущего шага)
   ./integrations/kaiten/scripts/kaiten-cards.sh check-item <card_id> <checklist_id> "Артефакт спайка проверен человеком"
   ```

9. Покажи пользователю итоговый summary:
   ```
   ✅ Spike завершён
   📄 Файл: <путь к файлу>
   📊 Рекомендация: <краткое описание решения>
   🔑 Класс сложности: <класс>
   🚧 Блокеры: <количество и краткое описание, или "нет">
   ```

## Важно

- Агенты пишут результаты **напрямую в файл** через Edit tool
- Агенты возвращают **только статус** (5-7 строк), не полные результаты
- Всегда проверяй `.repos-dir` перед клонированием — используй локальные репо, если доступны
- Спрашивай подтверждение перед клонированием репозиториев (только если нет локальной копии)
- Используй абсолютные пути к spike-файлу и репозиториям при вызове агентов
- **Source of truth артефакта — GitHub.** Не загружай .md файлы в Kaiten-карточку. Сначала запуши артефакт в GitHub, затем добавь ссылку на него в карточку.
- **Карточка спайка не должна теряться** — убедись, что она связана с другими карточками (родительская задача, задача на разработку и т.д.)
