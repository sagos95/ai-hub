# Spike from Meeting — встреча → markdown-артефакт со скриншотами

## Что делает

На вход: запись встречи (видео) + транскрипт с таймкодами.
На выходе: `<artifact-name>.md` в нужном пользователю формате + папка `pictures/` со скриншотами/GIF, привязанными к моментам обсуждения.

Главная идея — экономить время пользователя на ручной работе: вместо «прокрути на 12:40 и сделай скрин» — скилл сам находит ключевые моменты в транскрипте, дёргает кадры, **перепроверяет**, что на кадре действительно то, что обсуждают, и встраивает в документ с осмысленной подписью.

## Входные данные

`$ARGUMENTS` принимает 0–2 аргумента, любые комбинации:
- **Путь к видео-файлу** или **URL записи в Контур.Толк** (например `https://talk.kontur.ru/r/abc123`)
- **Путь к транскрипт-файлу** (опционально — если не указан, скилл попытается получить через API)
- **Формат документа**: `spike`, `recap`, `plan`, `instruction`, `custom` (опционально — иначе спросит)

Если пользователь скинул что-то одно — попроси недостающее. Не начинай работу без видео-источника.

## Этап 1. Получение транскрипта

### Режим A — пользователь дал файл транскрипта

Если в `$ARGUMENTS` есть путь к `.txt` / `.vtt` / `.srt` — используй его. Прочитай через Read, убедись что есть таймкоды формата `HH:MM:SS` или `MM:SS`. Если таймкодов нет — предупреди и спроси, есть ли версия с таймкодами.

### Режим B — пользователь дал URL Контур.Толк

Если в `$ARGUMENTS` URL вида `https://<tenant>.ktalk.ru/recordings/<id>` (например `<tenant>.ktalk.ru/recordings/...`) или `talk.kontur.ru/recordings/<id>`:

**Auth flow (cookies из браузера, без API-ключа):**

1. **Получи session token из Chrome cookies** — это самый быстрый путь, потому что Kontur.Talk использует SSO. Через chrome-devtools MCP (доступен в Claude Code):
   ```
   mcp__chrome-devtools__navigate_page → URL записи
   ↓ (Google SSO пропустит автоматом, если пользователь уже залогинен в Google в активном профиле)
   mcp__chrome-devtools__evaluate_script → 'document.cookie' → распарси `sessionToken=...`
   ```
   Если SSO попросит вручную выбрать аккаунт — попроси пользователя в браузере один раз пройти этап.

2. **Установи env:**
   ```bash
   export KTALK_SESSION_TOKEN="<значение sessionToken cookie>"
   ```

3. **Запусти `scripts/kontur-talk-transcript.sh <url>`** — он:
   - Распарсит URL (tenant + recording_id)
   - Вызовет `GET /api/recordings/v2/{id}/summary` с `Authorization: Session <token>`
   - Распарсит `transcriptionV2.tracks[].chunks[]` и выдаст плоский транскрипт в формате `HH:MM:SS<TAB>Имя<TAB>Текст` на stdout
   - Метаданные (title, duration, video_url, participants_count) в stderr как JSON
4. **Скачай видео:** `scripts/kontur-talk-video.sh <url> <output.mp4> [quality]` (по умолчанию — highest, обычно 900p).

**Карта API (верифицирована на 2026-05-12, реальный ktalk-tenant; работает на любом `*.ktalk.ru`):**

| Эндпоинт | Что возвращает |
|----------|----------------|
| `GET /api/recordings/{id}` | Метаданные: title, duration, conferenceKey, qualities[] (240p/900p), transcriptionUrl, participants[] |
| `GET /api/recordings/v2/{id}/summary` | `transcriptionV2.tracks[].chunks[]` (полный транскрипт), `shortSummaryV2`, `protocolV2` (AI-протокол) |
| `GET /api/conferenceshistory/v2/{conferenceKey}` | Chat-сообщения встречи, участники |
| `GET /recording-blob/{id}/{quality}` | MP4 видеопоток (с Range-поддержкой) |

Заголовок auth для всех: `Authorization: Session <sessionToken>` + `x-platform: web`.

> **Если SpeechCore (отдельный сервис транскриптов на `speechcore.kontur.host`) понадобится напрямую** — он требует отдельной аутентификации через SRS.Frontend.SpeechCore OIDC client. Обходить через `/api/recordings/v2/{id}/summary` — там всё уже доступно с тем же ktalk session token.

### Что делать с транскриптом дальше

Сохрани/скопируй транскрипт в рабочую папку артефакта (см. Этап 6 — структура папок) под именем `transcript.txt`. Это первоисточник, на который потом будут ссылаться таймкоды.

## Этап 2. Выбор формата документа

Если формат **не указан** в `$ARGUMENTS` — задай вопрос через `AskUserQuestion` с такими опциями:

| Формат | Когда выбирать | Структура |
|--------|----------------|-----------|
| **spike** | Исследовательская встреча по задаче из таск-трекера — нужен формальный спайк-отчёт | Job Story → гипотезы → варианты решения → рекомендация. Совместим с шаблоном `Spikes/YYYY-MM-DD_*.md` |
| **recap** | Регулярная встреча — нужно зафиксировать что обсудили / решили / кто что делает | Контекст → Обсудили → Решения → Action items (кто/что/когда) → Открытые вопросы |
| **plan** | Встреча со стейкхолдером — нужен план правок по категориям/приоритетам | Контекст → Категории работы с таблицами «что/где/зачем/таймкод» → Приоритеты P0/P1/P2 → TODO |
| **instruction** | Демонстрация пошагового процесса — нужно записать «как сделать X» | Цель → Шаги по порядку → каждый шаг = заголовок + что сделать + скриншот + проверка |
| **dual-artifact** | Нужны **два** артефакта: длинная статья + короткий пост-анонс (Slack/Time) | Два .md в одной папке: `article.md` (длинная статья для Buildin/Notion) + `slack-post.md` (5-10 строк, готов к копи-пасте в канал). Перекрёстные ссылки. Одна папка `pictures/` для обоих. |
| **custom** | Ни один пресет не подходит | Пользователь описывает желаемую структуру свободным текстом, скилл следует ей |

Сохрани выбор в локальную переменную/контекст — он определяет, **какие моменты в транскрипте интересны** (для `plan` — это смена экрана у стейкхолдера; для `instruction` — последовательность действий; для `recap` — кто что сказал и решения).

## Этап 3. Идентификация ключевых моментов

Прочитай транскрипт целиком. Найди **моменты, для которых нужен скриншот**:

- **Смена темы/экрана** — «сейчас покажу», «вот этот отчёт», «давай посмотрим X» — сигнал, что говорящий начал демонстрацию
- **Конкретная UI-деталь** — «здесь колонка такая-то», «эта метрика», «вот эта кнопка» — нужен кадр именно этого экрана
- **Сравнение/контраст** — «было так, а должно стать вот так» — потенциально GIF до/после
- **Решение / decision point** — «давай уберём этот столбец», «нужно добавить колонку X» — кадр контекста решения

Для каждого момента запиши кортеж: `(таймкод, описание_что_должно_быть_на_экране, к_какому_разделу_артефакта_привязан)`.

**Cколько скриншотов?** Ориентир по длительности — плотность скриншотов **падает** с длиной встречи (на длинных встречах темы повторяются, и одинаковые кадры не несут новой информации):

| Длительность | Целевое число | Плотность |
|--------------|---------------|-----------|
| 15-30 мин | 5-9 | ~1 кадр / 3-4 мин |
| 30-60 мин | 7-12 | ~1 кадр / 5-7 мин |
| 60-90 мин | 8-15 | ~1 кадр / 7-10 мин |
| 90+ мин | предложи пользователю разбить артефакт на 2-3 части по темам — иначе теряется фокус |

Подгоняй и под формат: `instruction` обычно требует больше (по скрину на шаг); `recap` / `dual-artifact` — меньше (только ключевые). Главный принцип сильнее любого числового ориентира: **один скриншот = одна новая идея / инструмент / UI**. Если соседние кадры показывают почти то же самое — оставь один.

## Этап 4. Извлечение + ВЕРИФИКАЦИЯ скриншотов

Это критичный этап — здесь скилл превращается из «тупого экстрактора» в «умного помощника».

### 4.1. Подготовка

```bash
mkdir -p <artifact_dir>/pictures
```

Проверь, что `ffmpeg` установлен:
```bash
command -v ffmpeg || brew install ffmpeg
```

### 4.2. Стратегия: interleaved (не batch)

**Делай по одному кадру, сразу верифицируй, сразу при необходимости переснимай — потом переходи к следующему.** Это эффективнее по контексту, чем «извлечь 9 кадров пачкой → потом 9 раз верифицировать → 3 раза переснять».

Используй `scripts/extract-frame.sh <video> <timecode> <output.png>`. Последовательно — `ffmpeg` seek по mp4 быстрый, но параллельно может упереться в IO и контекст агента.

Конвенция имён: `<N>-<M>_<topic-slug>_<HH-MM-SS>.png`, где `N.M` — номер раздела/пункта артефакта. Это нужно, чтобы файл сам подсказывал, к какому месту артефакта привязан.

### 4.3. Phrase-based timecode hints — выбирай умное смещение СРАЗУ

Таймкод фразы в транскрипте ≠ хороший таймкод для кадра. Зависит от типа фразы:

| Фраза в транскрипте | Типовое смещение | Почему |
|---------------------|------------------|--------|
| «**сейчас покажу**», «**пошарю экран**», «**давайте посмотрим**», «**вот сейчас открою**» | **+10..30s** | Спикер только заявил намерение. Реально окно найдено и демо показалось зрителю через 10-30 сек. До этого видна говорящая голова или ktalk-placeholder «You are sharing the screen». |
| «**вот этот отчёт**», «**эта вкладка**», «**вот здесь колонка X**», «**смотрите**» (без «сейчас») | **+1..3s** | Спикер уже показывает; нужно лишь дать кадру отрисоваться. |
| «**вот папочка Temp**», «**вот скачано**», «**вот это видно**» (указание на UI) | **±1s (ровно момент)** | Окно показа узкое — спикер часто листает дальше через 3-5 сек. |
| «**давай уберём X**», «**нужно добавить Y**» (решение по UI) | **−2..0s** | Контекст решения — это что было на экране ПЕРЕД самим решением. |
| «**покажу другое**», «**давайте перейдём к**» (переход) | **−2s** (последний кадр старого) или **+15s** (когда новое появилось) | Точка перехода: середина = пустой/переходный кадр. |

### 4.4. ВЕРИФИКАЦИЯ — самое важное

После каждой extract'а:

1. **Прочитай PNG через Read** (Read поддерживает изображения и покажет содержимое).
2. **Сравни с ожидаемым контекстом** из транскрипта:
   - Если в транскрипте «вот отчёт Заказы» — на скриншоте должна быть таблица заказов, а не главный экран / placeholder / рабочий стол.
   - Если в транскрипте «вкладка Сводка» — должна быть именно эта вкладка, а не Месяц/Команда.
3. **Распознай типовые провалы** (особенно на ktalk-записях):
   - **«You are sharing the screen» / «Stop sharing»** — это ktalk-placeholder, показанный самому презентующему ещё до старта реальной демонстрации (или после её окончания). **Сдвинь на +20..30s.** Если та же картинка — демо ещё не начато; ищи момент, когда говорящий уже описывает конкретный элемент UI.
   - **Только говорящие головы / gallery view** — share не активирован или временно скрыт. Сдвинь на +15s либо найди следующую фразу типа «вот».
   - **Слишком много панелей в кадре** — основной артефакт занимает <30% экрана. Это нормальный момент, но **не информативный**. Сдвинь на ±10s, ища кадр, где спикер скрыл другие окна.
   - **Картинка резкая, но не та** — спикер уже перелистнул на следующее. Сдвинь на **−5s** (раньше).
4. **Скорректируй и переизвлеки.** Лимит — 3 попытки на кадр. Если после 3-х не вышло — либо **дропни кадр** (бывают темы, у которых нет хорошего визуала), либо оставь с пометкой в подписи «*ближайший кадр демонстрации*».

### 4.5. Когда предложить GIF вместо скриншота

GIF имеет смысл если:
- Говорящий **навигирует** между вкладками («сейчас перейду на…», «открою настройки…»)
- Демонстрируется **анимация / интерактив** (раскрывающийся список, hover-эффект, drag-and-drop)
- Нужно показать **сравнение до/после** в одном экране (скролл/перелистывание)

В этих случаях после извлечения первого скриншота **спроси пользователя через `AskUserQuestion`**: «На таймкоде X идёт навигация — сделать вместо скрина короткий GIF (3–5 сек)?». Если да — `scripts/extract-gif.sh <video> <start_timecode> <duration_sec> <output.gif>`.

GIF выходит 1–3 МБ при разрешении 1280×720 и FPS=10 — приемлемо для репо. Не предлагай GIF массово (1–2 на встречу максимум), иначе раздуем репо.

### 4.6. PII / чувствительные данные

Если на скриншотах видны:
- Реальные имена + телефоны (PII штатных сотрудников)
- Имена клиентов, ID заказов
- Внутренние финансовые суммы

**Сначала спроси пользователя** — это репо приватное / публичное / какой контекст. Предложи через `AskUserQuestion` варианты: «замазать PII (рек.)» / «оставить как есть» / «замазать всё кроме UI-структуры».

Для замазывания используй `ffmpeg drawbox` с **полностью непрозрачным чёрным** прямоугольником (текст невосстановим). НЕ используй blur — текст всё ещё угадывается под низкоразмеренным размытием.

```bash
ffmpeg -y -i input.png -vf "drawbox=x=X:y=Y:w=W:h=H:color=black:t=fill" -update 1 output.png
```

**Не используй полупрозрачность** (`color=black@0.95` — текст просвечивает). Только `color=black`.

После редактирования **обязательно перечитай через Read** и убедись, что PII действительно скрыты, а не сдвинуты на пиксель. После подтверждения **удали unredacted-оригинал с диска** (`rm`), чтобы случайно не закоммитить.

## Этап 5. Сборка артефакта

Структура итогового `.md` зависит от выбранного формата (Этап 2). Общие правила для всех форматов:

### Цитирование таймкодов

В тексте таймкоды форматируй как inline-код: `` `00:12:40` ``. Если ссылаешься на цитату из транскрипта — приводи её в кавычках с указанием таймкода:
> `00:01:24` — «я бы вообще их убрала… возможно вывести отдельной строкой»

Если у пункта плана/раздела есть **диапазон обсуждения** + **ключевой момент** — указывай оба:
```
`00:05:35`–`00:08:23` (ключ: `00:06:37` — «формулировку здесь изменила…»)
```

### Встраивание скриншотов

```markdown
![Краткий alt-текст](./pictures/1-1_orders-report_00-02-55.png)
> **1.1** · MO → Отчёт «Заказы» · `00:02:55` — [подпись: что видно на кадре + почему это важно для пункта артефакта]
```

Подпись (blockquote `>`) должна:
- **Привязать кадр к разделу артефакта** (например «**1.1**» — пункт 1.1)
- **Назвать экран** (полный навигационный путь: «MO → Отчёт «Заказы»»)
- **Указать таймкод** (для перемотки видео)
- **Сказать ЧТО видно** + **почему смотрим** (не просто «вот скриншот», а конкретику: «видна метрика 0.86, посчитанная по часам всех курьеров, поэтому занижена»)

### Иерархия скриншотов в документе

- **Скриншоты внутри таблицы** не работают в markdown — встраивай их **под таблицей** в блоке «Скриншоты обсуждаемых экранов (как сейчас выглядят):»
- **Скриншоты внутри буллета** — отступай 2 пробела перед `![]` и `>`, чтобы они принадлежали буллету
- Для `instruction` формата — скриншот ставится **сразу после** заголовка шага

## Этап 6. Структура папок

Финальная раскладка артефакта:

```
<artifact_dir>/
├── <artifact-name>.md          # главный документ
├── transcript.txt              # копия исходного транскрипта
└── pictures/                   # все скриншоты + GIF
    ├── 1-1_<topic>_<timecode>.png
    ├── 1-2_<topic>_<timecode>.png
    ├── 2-1_<topic>_<timecode>.png
    └── nav-<topic>_<timecode>.gif
```

**Где создавать `<artifact_dir>`** — спроси пользователя через `AskUserQuestion`, если непонятно из контекста:
- В корне текущего репо рядом с проектной документацией (например `Projects/<project>/Meetings/<date>_<topic>/`)
- В `Spikes/YYYY-MM-DD_<name>/` (если формат `spike`)
- В произвольное место, которое укажет пользователь

**Не используй `<artifact-name>.md` с пробелами или кириллицей в имени файла** — это ломает относительные ссылки на скриншоты на некоторых рендерах markdown. Используй kebab-case на латинице.

## Этап 7. Финальная проверка перед сдачей

Прежде чем отдать артефакт пользователю — пройди чек-лист:

- [ ] Все скриншоты в `pictures/` встроены в `.md`? (нет «осиротевших» файлов)
- [ ] Все ссылки на скриншоты ведут на существующие файлы? (`ls pictures/` + grep `(./pictures/` в md)
- [ ] PII закрыта на всех скриншотах, где она была? Unredacted-оригиналы удалены с диска?
- [ ] Транскрипт скопирован в папку артефакта? Ссылка на него есть в «Связанные документы» секции?
- [ ] Таймкоды в подписях к скриншотам соответствуют моментам в транскрипте? (Прочитай транскрипт ещё раз на этих метках)
- [ ] Формат документа соответствует тому, что попросил пользователь? (Если был `plan` — есть таблицы по категориям и приоритеты? Если `instruction` — нумерованные шаги?)

Если всё ОК — выведи пользователю **краткое summary**: «Готово, артефакт здесь: `<path>`. N скриншотов, K GIF, M таймкодов. Видеоисточник: `<path/url>`».

## Формат-пресеты — подробно

### spike

Структура совместима с шаблоном спайк-исследования (`/ai-hub:spike`).
- Job Story: «Как **<кто>** в ситуации **<когда>** хочет **<что>**, чтобы **<зачем>**»
- Гипотезы → варианты решения → рекомендация
- Каждая фаза обсуждения встречи привязывается к фазе спайка
- Спайк-файл идёт в `Spikes/YYYY-MM-DD_<name>_<card_id>.md`

### recap

```markdown
# <Тема встречи> — <Дата>

## Контекст
[1-2 предложения: кто, зачем, какая проблема]

## Обсудили
- [пункт 1, опционально со скриншотом]
- [пункт 2]

## Решения
1. **Решение А** (`таймкод`) — что решили + кратко почему
2. **Решение Б** (`таймкод`)

## Action items
- [ ] **Кто** — что — к когда (`таймкод` обсуждения)

## Открытые вопросы
- [ ] Вопрос 1 — кому адресован
```

### plan

Идеально для встреч со стейкхолдером, где есть N категорий правок (пример — встреча с Ksenia 2026-04-27 по отчётности Outsource Delivery).

```markdown
# <Тема> — план правок после <Стейкхолдер>, <Дата>

## Контекст
[Источник: ссылка на транскрипт; что обсуждали]

### Категория 1 — <название>

**Корневая проблема.** [Почему категория важна]

| # | Где | Что сделать | Зачем | Таймкод |
|---|-----|-------------|-------|---------|
| 1.1 | <экран> | <правка> | <выгода> | `HH:MM:SS`–`HH:MM:SS` (ключ: `HH:MM:SS` — «цитата») |
...

**Скриншоты обсуждаемых экранов (как сейчас выглядят):**

![1.1](./pictures/1-1_<topic>_<timecode>.png)
> **1.1** · <экран> · `<timecode>` — [что видно + почему смотрим]

### Категория 2 — ...

## Приоритеты
1. **P0 — <категория>** — <правки>
2. **P1 — ...**

## Открытые вопросы / TODO
```

### instruction

```markdown
# Как <сделать X>

## Цель
[Что в итоге]

## Шаг 1. <Действие>

[Что сделать словами]

![Скриншот шага 1](./pictures/step-1_<topic>_<timecode>.png)
> `HH:MM:SS` — на экране: [что видно + куда нажимать]

**Проверка:** [как понять, что шаг получился]

## Шаг 2. ...

## Итог
[Финальное состояние]
```

### dual-artifact

Типовой паттерн: длинная статья для базы знаний (Buildin / Notion / репо) + короткий пост-анонс для Slack/Time. Структура:

```
<artifact_dir>/
├── article.md              # длинная версия (1500-2500 слов): контекст, разделы, выводы, action items
├── slack-post.md           # короткая версия (150-250 слов): TL;DR в виде Slack-блока
├── transcript.txt
└── pictures/               # одна папка скриншотов на обоих
```

**article.md** — структурируется аналогично `recap` или `plan` (зависит от типа встречи), плюс:
- В шапке: метаданные (дата, длительность, ссылка на запись, участники)
- TL;DR в начале (3-5 буллетов)
- В конце — секция «Связанные документы» со ссылкой на `slack-post.md`

**slack-post.md** — форматируется под прямой копи-паст в Slack/Time:
- Каждая строка начинается с `>` (Slack-стиль блокового цитирования делает текст компактнее)
- Опционально emoji-маркер (🤖, 📄, 🎥) в начале блока — Slack их рендерит
- Не более 5-10 строк информации
- В конце — ссылки: на полную статью (`article.md` относительный) и на запись
- В Slack/Time не будут видны embed-картинки из `pictures/` — поэтому **не вставляй сюда изображения**, оставь их в `article.md`

Пример формата slack-post.md:
```markdown
> 🤖 *<Заголовок встречи> — <одна строка контекста>*
>
> *Что обсудили:*
> • Пункт 1
> • Пункт 2
> • Пункт 3
>
> *Главный вывод:* <одна сильная фраза>
>
> 📄 Полная статья: [article.md](./article.md)
> 🎥 Запись: <URL>
```

### custom

Спроси пользователя 1–3 наводящими вопросами: «Какие разделы должны быть?», «Нужны ли таблицы?», «Какая степень формальности?». Сгенерируй структуру по ответам.

## Скрипты-помощники (в `scripts/`)

- **`extract-frame.sh <video> <HH:MM:SS> <output.png>`** — один кадр через `ffmpeg -ss ... -frames:v 1`. Без аудио, качество `-q:v 2`.
- **`extract-gif.sh <video> <HH:MM:SS> <duration_sec> <output.gif>`** — короткий GIF, 1280×720, FPS=10, оптимизация через двухпроходный palette filter.
- **`kontur-talk-transcript.sh <recording_url>`** — забор транскрипта из Контур.Толк API. Читает `KTALK_SESSION_TOKEN` из env (значение cookie `sessionToken` из Chrome). Возвращает транскрипт `HH:MM:SS<TAB>Имя<TAB>Текст` в stdout + метаданные JSON в stderr. Эндпоинты верифицированы 2026-05-12 на реальном ktalk-tenant'е, должны работать на любом `*.ktalk.ru`.
- **`kontur-talk-video.sh <recording_url> <output.mp4> [quality]`** — скачивает MP4 видеозаписи. По умолчанию highest quality (обычно 900p). Использует тот же `KTALK_SESSION_TOKEN`.

## Что НЕ делать

- **Не пиши** скриншоты в корень артефакта — только в `pictures/`. Иначе теряется чистота.
- **Не используй** имена файлов с пробелами / кириллицей для `.md` или `.png` — относительные ссылки сломаются на некоторых рендерах.
- **Не вставляй** скриншот без подписи. Голая картинка без контекста бесполезна.
- **Не извлекай** один и тот же кадр без верификации. Один цикл «извлёк → прочитал → проверил соответствие транскрипту» обязателен для каждого скриншота.
- **Не извлекай пачкой 9 кадров → потом 9 раз верифицируй.** Это раздувает контекст. Делай interleaved: один кадр — verify — если плохо, сразу пересними — переходи к следующему (см. 4.2).
- **Не игнорируй фразовый контекст таймкода.** «Сейчас покажу X» в `00:01:30` ≠ X на экране в `00:01:30`. Перед extract'ом сверься с таблицей в 4.3.
- **Не блюрь полупрозрачным боксом** — текст просвечивает. Только сплошной чёрный.
- **Не оставляй** unredacted-копии PII на диске после редактирования — `rm` сразу.

## Если что-то идёт не так

- **`ffmpeg` падает** — проверь, что путь к видео без кавычек/пробелов; если есть пробелы, оборачивай переменную в `"$VAR"`.
- **Скриншот всегда показывает плейсхолдер «You are sharing screen»** — это значит, что в записи видна сторона **зрителя**, а не презентующего. У Контур.Толк такое бывает на крупных встречах. Сообщи пользователю — придётся искать другую запись с записанным экраном докладчика.
- **Транскрипт без таймкодов** — спроси у пользователя, есть ли версия с временными метками. Если нет — придётся либо запускать speech-to-text заново, либо собирать таймкоды вручную.
- **Очень длинная встреча (>1ч)** — не пытайся прогнать всё за один раз. Спроси пользователя, какой **интервал** интересен, либо предложи разбить на 2–3 артефакта по темам.
- **Kontur.Talk не пускает без логина в браузере** — Google SSO для `*.ktalk.ru` обычно прыгает молча, но если выбрасывает на форму выбора аккаунта — пользователь должен кликнуть свой аккаунт один раз в браузере. Дальше cookie доступна для всех скриптов.

## Lessons from past runs

История запусков скилла. Помогает не повторять ошибки и видеть, какие настройки работают на каких типах встреч.

### 2026-04-27 · Встреча с Ksenia (Outsource Delivery reports) · формат `plan`

- **Длительность:** ~30 мин · **Скриншотов в финале:** 9 (плотность ~1 / 3.3 мин — высокая, потому что встреча шла по структуре «9 экранов с правками»)
- **Источник:** локальные файлы (видео + транскрипт), API не нужен
- **Что работало хорошо:** структура `plan` идеально легла на 3-категориальное обсуждение (производительность / скорость / зарплата); таймкоды как привязка к видео; PII-редакция через `drawbox` (4 кадра содержали имена + телефоны курьеров)
- **Что было трением:** 2 кадра захватили «You are sharing screen» плейсхолдер — пришлось переснимать со смещением `+10..15s`

### 2026-05-07 · WorkShop AI MAGIC · формат `dual-artifact`

- **Длительность:** 65 мин · **Скриншотов в финале:** 8 (плотность ~1 / 8 мин — нормально для длинной встречи; темы повторялись и одинаковые кадры не несли новой информации)
- **Источник:** Kontur.Talk URL → API забор через `kontur-talk-transcript.sh` + `kontur-talk-video.sh` (cookies из Chrome SSO)
- **Что работало хорошо:** auto-detect tenant'а из URL; SSO прошёл молча через chrome-devtools; `transcriptionV2` от ktalk-API сразу даёт chunks с ms-таймкодами; `dual-artifact` подошёл точно под запрос «статья + пост для Slack»
- **Что было трением:** из 9 первых кадров 1 пришлось дропнуть (placeholder, не помогло сместить); 2 переснять (`+30s` для одного, `+15s` для другого). Если бы я использовал phrase-based hints из 4.3 с самого начала — было бы 0-1 переснимок.

### Принципы, проверенные на практике

- **Один скриншот = одна новая идея.** Если два соседних кадра показывают почти то же самое — оставь один.
- **Placeholder «You are sharing the screen» появляется в ktalk-записях** в момент начала и конца шаринга. Всегда сдвигай `+20s` от фразы «сейчас покажу».
- **Цитата в подписи под скриншотом сильнее, чем длинная alt-тема.** Прямая речь спикера в `>`-блоке делает кадр «живым».
- **На длинной встрече дроп лучше, чем плохой кадр.** Если за 3 попытки не нашёл хорошего — у этой темы просто нет визуала, опиши текстом.
